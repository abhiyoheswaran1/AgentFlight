## Scope

This dogfood pass exercised the released v0.2.0 core workflow against the AgentFlight repository. It did not add product features, bump versions, tag, or release anything.

## Commands Run

```bash
npx agentloopkit@latest create-task --title "Dogfood AgentFlight v0.2.0 core workflow" --type docs ...
node dist/cli.js start --task "Dogfood AgentFlight v0.2.0 core workflow"
node dist/cli.js verify -- npm run typecheck
node dist/cli.js verify -- npm run lint
node dist/cli.js verify -- npm test
node dist/cli.js verify -- npm run build
node dist/cli.js status
node dist/cli.js report
node dist/cli.js replay
node dist/cli.js resume
node dist/cli.js doctor
```

## What Felt Good

- The happy-path workflow is coherent: start, make a change, verify, status, report, replay, resume, doctor.
- `agentflight verify -- npm test` feels natural for users who know the conventional `--` command separator.
- Verification evidence is persisted in both the current session JSON and per-run stdout/stderr files.
- `status` clearly shows `4 passed, 0 failed`, no configured gaps, and `Ready for review`.
- `doctor` is useful and concise when the repository is configured correctly.
- The Markdown report is useful enough for an internal handoff because it lists changed files, risk, evidence, and output paths.

## What Felt Confusing

- A docs-only dogfood pass was reported as `medium` risk because `.agentloop/state.json` was categorized as `unknown`.
- The generic risk reason, `Risk is based on changed file categories.`, does not explain why the session is medium risk.
- The report says `Next Action: Generate a report and hand off for review` even inside the generated report. After report generation, that recommendation feels stale.
- The report showed `ProjScan: available 0.9.2`, while `npx projscan@latest --version` reported `4.3.1`. That looks like adapter version detection drift.

## What Felt Too Verbose

- The replay and report include long evidence paths repeatedly. This is useful proof, but it dominates the visual output.
- `resume` can become long quickly because it lists every changed file. In this pass there were only three files, so it was acceptable.

## What Felt Too Hidden

- `agentflight verify` confirms pass/fail and prints stdout, but it does not tell the user where stdout/stderr evidence was saved.
- `report` and `replay` print only the artifact path. They do not suggest opening the replay or show a relative path that is easy to copy.
- The session file is understandable once opened, but the CLI does not surface `.agentflight/current/session.json` as the source of truth.

## Verify Syntax

The syntax is acceptable:

```bash
agentflight verify -- npm test
```

The tradeoff is repetition. Running four proof commands requires four `agentflight verify -- ...` invocations. The configured no-argument mode should become the preferred workflow once config seeding is clearer.

## Status

`status` answers the core questions:

- What changed.
- What risk exists.
- What proof exists.
- Whether review is ready.
- What to do next.

The main friction is risk clarity. A docs-only change should not feel medium risk unless the explanation names the exact reason.

## Report

The report is useful enough to share internally. It is honest about verification evidence and includes output paths.

Recommended improvement: make the next action context-aware. After a report exists, the report should not tell the user to generate a report.

## Replay

The replay is functional and clean enough for local review. It is not yet screenshot-worthy as a viral artifact.

Reasons:

- Long evidence paths dominate the verification cards.
- Visual hierarchy is plain and utilitarian.
- There is no summary strip like `4 passed / 0 failed / Ready for review`.

## Resume

The resume prompt is useful enough to hand to Codex or Claude for continuation. It includes task, state, changed files, risks, verification gaps, next action, and guardrails.

Recommended improvement: when proof is complete and a report/replay exists, point the next agent to the generated artifacts instead of saying to generate a report.

## Evidence And Session Files

The evidence paths and session files are understandable once found:

- `.agentflight/current/session.json`
- `.agentflight/evidence/<session-id>/verification-N.stdout.txt`
- `.agentflight/evidence/<session-id>/verification-N.stderr.txt`

Recommended improvement: `agentflight verify` should print a short `Evidence saved:` line for each run.

## Failed Verification Behaviour

This pass did not intentionally add a failing verification run to the active repository session because the goal was to keep the dogfood session ready for review.

Expected v0.2.1 improvement based on the current UX:

- On failure, `agentflight verify` should print the stderr evidence path and a clear next action.
- `status` should highlight the failed command before lower-risk summary text.
- `resume` should include the exact failed command to rerun after fixing.

## Bugs Found

- ProjScan adapter/version display appears incorrect in the report: it showed `0.9.2` while `npx projscan@latest --version` showed `4.3.1`.
- `.agentloop/state.json` is categorized as `unknown`, which pushes a docs-only workflow to `medium` risk without a useful explanation.

## Recommended v0.2.1 Fixes

1. Fix ProjScan adapter version detection.
2. Improve risk categorization for `.agentloop/` files or generated workflow state.
3. Make `status` risk reasons more specific for mixed docs/tooling changes.
4. Print evidence paths directly from `agentflight verify`.
5. Make report/resume next actions context-aware after report/replay generation.
6. Add a summary strip to replay: risk, changed files, passed/failed proof, readiness.
7. Consider seeding `.agentflight/config.json` with detected proof commands during `init` or `start`, or make the configured `agentflight verify` path clearer.

## v0.2.1 Patch Follow-Up

Addressed locally:

- ProjScan and AgentLoopKit adapters now prefer repo-local `node_modules/.bin` binaries before PATH-global commands, fixing the stale global ProjScan version shown during dogfooding.
- `.agentloop/` workflow artifacts are categorized as low-risk docs so a docs-only dogfood pass does not become medium risk because of AgentLoopKit state.
- `agentflight verify` prints `Evidence saved:` paths for stdout and stderr.
- Failed verification next actions include the exact command to rerun.
- Report, replay, and resume next actions avoid telling the user to generate a report after proof is already ready.
- Replay now has a compact summary strip for risk, changed files, proof counts, and readiness.

Deferred deliberately:

- Seeding `.agentflight/config.json` with proof commands.
- Adding new commands such as `snapshot`.
- JSON output, GitHub PR comments, cloud sync, login, billing, or Pro/Team features.
- Larger replay visual redesign and long evidence-path compaction.
