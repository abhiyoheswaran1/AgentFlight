# AgentFlight v0.5.0 Dogfood Findings

Dogfood date: 2026-06-17

Published package tested: `agentflight@latest`, confirmed as `0.5.0`.

## Summary

AgentFlight v0.5.0 is a meaningful trust improvement over the earlier report
and replay flow. The Markdown report and HTML replay both put the failing
verification excerpt in the right place, prefer stderr over stdout, keep the
excerpt short, and safely escape HTML in replay output. The redesigned replay
ledger is easier to scan than the older timeline-heavy view because the pass and
fail runs sit together with exit status, runtime, evidence paths, and the failure
excerpt.

The main release-quality issue is that the inline terminal output from
`agentflight verify` did not use the same stderr-preferred excerpt behavior. In
both tested repos the controlled failure wrote useful lines to stderr and noise
to stdout, but the immediate CLI output showed the stdout noise lines. The
persisted Markdown report and HTML replay showed the useful stderr excerpt.

Based on this pass, a small v0.5.1 patch is warranted before broader v0.6.0
planning.

## Failure-Path Findings

Controlled failure command used in both tested repos:

```bash
npx --yes agentflight@latest verify -- node -e 'console.log("DOGFOOD_STDOUT_NOISE: this stdout line should not dominate the excerpt"); console.log("DOGFOOD_STDOUT_NOISE: second stdout line"); console.error("DOGFOOD_EXPECTED_STDERR: controlled failure line 1"); console.error("HTML_ESCAPE_CHECK: <script>alert(\"agentflight-dogfood\")</script> & <b>bold</b>"); console.error("DOGFOOD_EXPECTED_STDERR: final actionable failure"); process.exit(42)'
```

Evidence:

- AgentFlight failure exited `42` in `252ms`.
- fifa-predictor failure exited `42` in `324ms`.
- In both repos, `verification-2.stdout.txt` contained only the two
  `DOGFOOD_STDOUT_NOISE` lines.
- In both repos, `verification-2.stderr.txt` contained the three useful
  `DOGFOOD_EXPECTED_STDERR` and `HTML_ESCAPE_CHECK` lines.

Findings:

- Markdown report excerpt: correct. It preferred stderr and showed exactly the
  three useful lines.
- HTML replay excerpt: correct. It preferred stderr and showed exactly the
  three useful lines.
- Inline terminal excerpt: incorrect. The immediate `verify` output showed the
  stdout noise lines instead of stderr.
- Excerpt length: right for report/replay. It was short enough to scan and did
  not bury the actual failure.
- Excerpt noise: low in report/replay, high in inline terminal output because
  stdout won there.
- HTML replay escaping: safe in the generated HTML. The replay stored the probe
  as `&lt;script&gt;...&lt;/script&gt;`, Playwright reported `document.scripts.length`
  as `0`, and no dialog executed when the replay was opened.

## Replay Ledger UX Findings

- The ledger improves review speed. The failure entry is easy to find because
  it is marked `FAIL`, includes exit code and duration, and displays the stderr
  excerpt inline.
- The pass entry is appropriately lower emphasis. It keeps evidence files and
  successful output behind a `details` disclosure.
- The sticky readout band helps orientation while scrolling. At `scrollY: 2100`
  Playwright confirmed `.summary-grid` was `position: sticky` with `top: 0`.
- The sticky band can overlap jumped-to section headings. When Playwright
  scrolled `Verification Evidence` into view, the heading landed underneath the
  sticky band. The entries remained readable, but a future scroll-margin polish
  would help.
- Long suggested proof commands in Review Focus and Proof Gaps delay access to
  the ledger. The ledger is strong, but in these controlled failures it appeared
  below long command text.

## Report Usefulness

- The Markdown report is useful as a proof artifact. It includes timeline,
  changed files, risk, verification evidence, failure excerpt, review focus,
  proof gaps, readiness, recommendation, and tooling.
- Report/replay/status/resume agreed after the controlled failure on changed
  files, latest snapshot, `1 passed / 1 failed`, blocking proof gap, and blocked
  readiness.
- Status and resume do not include the failure excerpt, while report and replay
  do. That is acceptable, but the report/replay are the faster review surfaces
  for failures.
- The fifa-predictor Markdown report became noisy in the Tooling section because
  AgentLoopKit was unavailable/configured as failing and the report embedded a
  long doctor failure block. That is factual but too much detail for a proof
  report.
- First-run generated files still affect report usefulness. In fifa-predictor,
  `.projscan-memory/memory.json` ranked above `.agentflight/config.json` as an
  unknown file and added an info proof gap.

## Review Intelligence Consistency

- Review Intelligence remained conservative and consistent after failure:
  readiness changed to `Blocked by failed verification` in status, report,
  replay, and resume.
- Proof gaps correctly included a blocking failed-command gap.
- The latest snapshot shown by status and resume was the controlled failure

- The failure excerpt does not conflict with Review Intelligence, but Review
  Focus and Proof Gaps repeat the full failing command heavily. That repetition
  makes the important excerpt harder to reach.

## Bugs Found

1. Inline `agentflight verify` failure output does not prefer stderr when stderr
   is present. Report and replay do prefer stderr.
2. Replay section anchors or programmatic `scrollIntoView` can land behind the
   sticky readout band.
3. The Markdown report can embed an overly long AgentLoopKit doctor failure in
   the Tooling section when AgentLoopKit is unavailable or not configured for the
   repo.
4. First-run ProjScan memory still appears as a changed file unless
   `.projscan-memory/**` is ignored in AgentFlight config.
5. Tool availability messaging was inconsistent in fifa-predictor:
   `init` reported ProjScan unavailable, while the later session/report path
   detected or invoked ProjScan and generated `.projscan-memory/memory.json`.

## Recommended v0.5.1 Patch Items

1. Fix inline `verify` terminal output to use the same stderr-preferred excerpt
   selection as Markdown report and HTML replay. Patch candidate status:
   implemented in the v0.5.1 worktree.
2. Compact long suggested proof commands in Review Focus, Proof Gaps,
   Recommendation, and PDF output. Consider showing the command label plus
   evidence path, with full command behind details in HTML. Patch candidate
   status: dense review surfaces now render compact command labels; HTML keeps
   the full command in a local title attribute.
3. Trim Tooling output in reports when AgentLoopKit is unavailable or failing.
   A one-line status plus the actionable next step is enough. Patch candidate
   status: Markdown reports summarize AgentLoopKit doctor/unavailable warnings.
4. Add or improve default handling for `.projscan-memory/**` in first-run
   sessions so generated memory does not outrank actual review files. Patch
   candidate status: guidance remains suggestion-only and is clarified in
   `docs/development/changed-file-filters.md`.
5. Add scroll margin or anchor offset for replay sections below the sticky
   readout band.

## v0.6.0 Candidates

- Better first-run workspace hygiene: preview generated files, explain which
  files are expected, and offer an explicit cleanup or ignore-guidance workflow.
- A richer replay review mode with collapsible Review Focus commands, direct
  jump links to failed runs, and less repetition between proof gaps and
  recommendation.
- Named verification profiles so reports can use short labels instead of
  repeating long shell commands everywhere.
- Consistent tool availability and fallback reporting across `init`, `start`,
  `status`, `report`, and `doctor`.
- Optional reviewer export modes: compact Markdown, full Markdown, print/PDF,
  and agent-resume prompt.

## Recommendation

Ship a v0.5.1 patch for the inline stderr-preferred excerpt mismatch and report
noise. Keep v0.6.0 focused on workflow polish around first-run hygiene,
review-mode navigation, and verification profile ergonomics rather than new
product surface area.
