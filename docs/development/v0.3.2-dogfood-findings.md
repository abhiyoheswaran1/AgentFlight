# AgentFlight v0.3.2 Dogfood Findings

## Summary

Dogfooding used the published `agentflight@latest` package, confirmed as `0.3.2`,
against three local repos:

- AgentFlight
- ProjScan
- fifa-predictor, used as the real app repo

AgentLoopKit and TokenTrace were locally available, but both had pre-existing
dirty worktrees. They were inspected and intentionally not modified during this
pass.

The core workflow worked end to end:

```bash
npx --yes agentflight@latest init
npx --yes agentflight@latest start --task "Dogfood AgentFlight v0.3.2 in this repo"
npx --yes agentflight@latest verify -- <repo command>
npx --yes agentflight@latest snapshot --note "Dogfood checkpoint after verification"
npx --yes agentflight@latest status
npx --yes agentflight@latest report
npx --yes agentflight@latest replay
npx --yes agentflight@latest resume
npx --yes agentflight@latest doctor
```

The strongest finding: AgentFlight is useful once a repo already ignores its
runtime files, but first-use dogfooding in a fresh repo makes `.agentflight/`
evidence and generated ProjScan memory show up as changed files. That pollutes
risk, report, replay, and resume output.

## Cross-Repo Observations

- The published package works from `npx --yes agentflight@latest` and reports
  `0.3.2`.
- `verify` is the strongest part of the product: it turns local commands into
  durable evidence with paths the user can inspect.
- `status` gives a clear review-readiness answer when changed files are clean.
- `resume` is practical for handing work to another coding agent.
- First-use setup in repos without AgentFlight ignore rules creates noisy
  changed-file lists.
- Runtime files should probably be excluded from AgentFlight's own changed-file
  analysis even if the target repo has not ignored them yet.
- The replay artifact is useful, but it currently feels more like a generated
  report page than a polished developer cockpit.

## Bugs Found

- Interrupted verification can leave an orphan `verification_started` event
  without a failed/interrupted verification result.
- AgentFlight counts `.agentflight/` runtime files in changed-file analysis when
  the target repo has not ignored them. Patch status: addressed in the v0.3.3
  candidate by excluding runtime session, report, current, and evidence paths
  while keeping `.agentflight/config.json` visible.
- ProjScan adapter/fallback behavior can create `.projscan-memory/memory.json`,
  which then appears as a user code change.
- Availability messaging can differ across `init`, `start`, and `doctor`
  because they use different detection/fallback paths.

## UX Friction

- `init` output such as `Created: 0` is terse and slightly awkward.
- Long-running `verify` commands provide no heartbeat or elapsed-time feedback.
- `status` can become noisy because evidence files and handoff files are shown
  in changed files.
- `resume` inherits changed-file noise, which weakens the handoff prompt.
- AgentLoopKit task creation is useful for dogfooding, but in AgentFlight's own
  repo it created tracked side effects that had to be manually cleaned.

## Feature Ideas

- Filter AgentFlight runtime paths from changed-file/risk/report/replay/resume
  analysis by default. Patch status: addressed in the v0.3.3 candidate for
  `.agentflight/sessions/`, `.agentflight/reports/`, `.agentflight/current/`,
  and `.agentflight/evidence/`.
- Offer to add or validate `.gitignore` entries during `agentflight init`.
- Add an interrupted verification status, for example `interrupted`, with exit
  code/signal metadata where available.
- Add progress feedback for long-running verification commands.
- Make tool availability detection consistent across `init`, `start`, and
  `doctor`.
- Improve replay visual hierarchy for developers: denser timeline, clearer
  evidence cards, better file grouping, and stronger review-readiness emphasis.

## v0.4.0 Recommendation

Based only on this dogfood evidence:

1. **E. Better replay/report UX**: the product already records useful evidence,
   but the generated artifacts should feel more like a polished developer
   control room.
2. **D. Deeper ProjScan risk enrichment**: runtime/generated files polluted risk
   scoring, and ProjScan fallback artifacts need clearer handling.
3. **A. Review intelligence**: status/report/resume should translate evidence
   and risk into sharper reviewer guidance once changed-file noise is filtered.

JSON/CI integration and PR comment generation still matter, but the current
dogfood evidence points first to making local artifacts clearer and less noisy.
